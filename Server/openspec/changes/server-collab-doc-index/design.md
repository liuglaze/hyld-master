## Context

当前服务端仓库的 `CLAUDE.md` 已覆盖架构细节，但缺少面向外部协作方（客户端开发、联调同学、新接手同学）的“任务型索引”。
现状问题主要有三点：
1. 信息按模块组织，不按协作场景组织，外部同学定位入口慢。
2. OpenSpec 配置虽有流程约束，但缺少跨端影响、兼容矩阵、联调证据等强约束。
3. Server 层与根目录 OpenSpec 配置存在上下文落差，跨目录协作容易丢失统一规范。

该变更属于文档与规范治理，不改变运行时代码路径，不引入新依赖。

## Goals / Non-Goals

**Goals:**
- 将 `Server/CLAUDE.md` 重构为“对外协作入口文档”，前置快速索引与联调入口。
- 在 `Server/openspec/config.yaml` 增加跨端协作 artifact 规则，约束 proposal/design/specs/tasks 的输出质量。
- 在根 `openspec/config.yaml` 提供最小上下文与规则，使跨目录协作保持一致。
- 让后续需求在 OpenSpec 中天然产出可交付给客户端的证据（影响摘要、兼容矩阵、联调步骤）。

**Non-Goals:**
- 不修改 protobuf 定义、Controller 路由或 DAO 行为。
- 不改变 TCP/UDP 协议载荷与端口。
- 不解决现有运行时并发问题（例如 UDP 单例回调串扰），仅要求文档中必须披露并评估该风险。

## Decisions

### Decision 1: 采用“先导航后细节”的 CLAUDE.md 结构
- 方案：将“读者范围、60 秒上手、场景索引、联调清单”前置，再放架构分层与风险。
- 原因：外部协作方最常见需求是定位入口和联调，不是先读全量架构背景。
- 备选：维持现有结构，仅追加附录索引。
- 不选原因：附录可见性低，无法显著降低首次定位成本。

### Decision 2: 在 OpenSpec rules 中引入跨端产物硬约束
- 方案：对 proposal/design/specs/tasks 分别增加必填约束（跨端影响摘要、兼容矩阵、联调证据、可观测性字段）。
- 原因：避免“只写实现思路，不写协作证据”的工件质量问题。
- 备选：仅在团队约定中口头要求。
- 不选原因：不可执行、不可检查，容易回退为个人习惯。

### Decision 3: 保持运行时零侵入，仅落地文档与配置
- 方案：本次仅改 `CLAUDE.md` 与 `config.yaml`，不触碰业务代码。
- 原因：用户诉求是协作可见性与索引优化，属于治理层问题，应最小化变更面。
- 备选：同步改造日志系统和协议注释。
- 不选原因：扩大范围，增加回归风险，不满足“快速落地”目标。

## Risks / Trade-offs

- [风险] 文档更新后未被持续维护，索引很快失真
  → [缓解] 在 OpenSpec `tasks` 规则中加入“文档一致性检查”子任务。

- [风险] 规则过严导致提案撰写成本上升
  → [缓解] 仅要求最小可验证证据（影响摘要 + 兼容矩阵 + 联调步骤），避免过度模板化。

- [风险] 根目录与 Server 目录规则产生冲突
  → [缓解] 根目录只保留通用协作底线，Server 目录保留游戏服务器特有约束。

## Migration Plan

1. 更新 `Server/CLAUDE.md` 为对外协作索引结构。
2. 更新 `Server/openspec/config.yaml` 的 artifact 规则。
3. 补齐根 `openspec/config.yaml` 最小上下文与通用规则。
4. 运行 `openspec status --change server-collab-doc-index` 确认工件状态。

回滚策略：
- 文档/配置回退到变更前版本即可，无数据迁移或运行时副作用。
