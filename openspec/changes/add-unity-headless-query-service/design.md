## Context

当前战斗服务器已经具备权威帧循环、输入缓冲、玩家位置推进、服务端子弹推进以及命中后扣血/死亡广播能力，关键入口位于 `Server/Server/Battle.cs:320`、`Server/Server/Battle.cs:493`、`Server/Server/BattleController.Bullets.cs:164` 与 `Server/Server/BattleController.Bullets.cs:266`。现阶段的缺口不是战斗主循环，而是服务端缺少一份可查询的场景物理世界：没有地图 collider、没有墙体或障碍物阻挡、也没有统一的路径/范围查询接口，因此所有命中近似建立在“无遮挡世界 + 玩家位置半径判定”之上。

本变更引入一个与现有权威服并行运行的 Unity headless 查询服务。主服务器继续保持战斗状态真源，负责输入消费、权威状态推进、伤害落账与广播；Unity 进程只维护服务器专用的碰撞世界和玩家代理体，并通过本机 IPC 提供同帧返回的空间查询。第一阶段目标是补齐投射物路径首碰撞、遮挡判定和范围查询，不改写现有客户端协议语义，也不重构玩家移动碰撞与完整刚体模拟。

## Goals / Non-Goals

**Goals:**
- 为现有权威战斗服补充一层本机低延迟的 Unity 物理查询能力，而不是替换主服务器。
- 建立清晰的职责边界：主服务器负责战斗规则与状态真源，Unity headless 只负责空间查询。
- 让 `TickServerBullets(...)` 中的投射物推进从“离散位置 + 距离判定”升级为“路径 sweep 查询 + 首碰撞结果消费”。
- 提供统一的查询类型，覆盖第一阶段所需的 `ProjectileCast`、`LineOfSight`、`OverlapSphere`。
- 保证查询服务异常时主服务器仍可回退到旧判定路径，避免阻断战斗房间。

**Non-Goals:**
- 第一阶段不把整个战斗服迁移到 Unity Dedicated Server。
- 第一阶段不让 Unity 查询服务承担伤害结算、Buff、击杀、胜负等规则裁决。
- 第一阶段不实现完整历史物理世界回放，不要求用 headless 对历史帧障碍状态做回溯模拟。
- 第一阶段不重构玩家移动碰墙、滑墙、导航或角色控制器。
- 第一阶段不要求每颗子弹在 Unity 查询服中作为长期存在的 GameObject 运行。

## Decisions

### 1. 采用“双进程 + 本机 IPC”，而不是把主战斗逻辑直接迁入 Unity

**Decision**
- 保留现有主服务器进程作为战斗真源。
- 新增一个同机运行的 Unity headless 查询进程，使用本地 IPC 与主服务器通信。

**Rationale**
- 当前主服务器已经拥有稳定的帧循环、权威位置推进和命中落账能力，缺的是场景物理层，不需要推翻现有战斗骨架。
- 双进程方案可以最小化改造面，降低首版风险，并保留平滑回退能力。
- 同机 IPC 能满足同帧查询需求，且比跨机器 RPC 更容易控制时延。

**Alternatives considered**
- **直接迁移为 Unity 逻辑服**：终局可能成立，但当前成本过高，且会把主服务器网络、会话与战斗规则一起迁移，超出第一阶段目标。
- **远程 HTTP/TCP 查询服务**：实现简单，但时延和抖动更难保证，不适合绑定 16ms 战斗帧预算。

### 2. 查询服务只维护“镜像物理世界”，不自己推进战斗状态

**Decision**
- 主服务器继续推进玩家位置和子弹数据。
- 每帧由主服务器把当前权威玩家状态同步到查询服务，查询服务只更新代理体并执行查询。

**Rationale**
- 这样可以避免“双真相”：不会出现主服务器和 Unity 两边都在推进位置、互相漂移的问题。
- 现有 `Battle.cs:493` 已经定义了权威位置推进逻辑，复用这份结果最稳。
- 代理体镜像方案对第一阶段只做空间查询已经足够。

**Alternatives considered**
- **查询服务也自己跑帧推进**：会引入两套状态机和帧同步问题，调试复杂度明显升高。

### 3. 第一阶段仅同步必要状态：静态地图 collider + 玩家代理体 + 动态障碍事件

**Decision**
- 静态地图碰撞体由 Unity 查询服在房间创建时加载服务器专用场景。
- 动态同步对象限制为玩家代理体，以及确有需要的动态障碍体状态。
- 子弹默认不在查询服中长期实例化，而是按请求即时做 cast/overlap。

**Rationale**
- 你们当前最痛的是子弹遮挡、路径首碰撞和范围判定，而不是完整投射物实体管理。
- 即时查询比“每颗子弹一个 GameObject”更轻，且能直接挂接现有 `ServerBullet` 数据结构。
- 场景不重时，这样更容易保证 IPC 与查询耗时压进同帧内。

**Alternatives considered**
- **在 Unity 查询服里长期维护所有子弹对象**：更接近完整模拟，但首版复杂度高，而且职责会向“第二个战斗服”滑移。

### 4. 统一查询协议拆成“状态同步”和“查询请求”两层

**Decision**
- 每帧先发送一次 `SyncPlayers`（必要时还有 `SyncDynamicObstacles`）。
- 然后在同一帧内发送多个查询请求，例如 `ProjectileCast`、`LineOfSight`、`OverlapSphere`。
- 查询返回值只描述空间结果：首个命中对象、命中点、法线、距离、阻挡类型。

**Rationale**
- 避免每颗子弹请求重复携带整份玩家状态，降低 IPC 包体与序列化开销。
- 让查询协议语义更稳定，后续可以复用到技能、机关或环境判定。
- 空间查询结果与战斗规则解耦，避免把伤害结算散进 Unity 查询服。

**Alternatives considered**
- **每个请求都带完整战场状态**：简单但冗余，随着查询量上升会明显放大 IPC 成本。
- **查询服直接返回扣血/死亡结果**：职责污染，不利于主服务器保持规则真源。

### 5. 主战斗循环中的接入顺序固定为“先同步，再查询，再落账”

**Decision**
- 在当前帧中，主服务器先完成输入消费和玩家位置推进。
- 然后同步玩家代理体到 Unity 查询服。
- 子弹推进时，对 `oldPos -> newPos` 执行 `ProjectileCast`；命中结果返回后，再由主服务器执行扣血、死亡、HitEvent 生成与广播。

**Rationale**
- 这样能保证查询永远基于当前帧的权威玩家位置，而不是旧镜像状态。
- 将 `TickServerBullets(...)` 作为主要接入点，可以以最小改动获得最大的收益。
- 让现有 `CheckBulletCollision(...)` 逐步退化成“命中结果落账层”，职责更清晰。

**Alternatives considered**
- **异步跨帧查询**：会导致命中帧号错位，使现有 `frameid` 与 HitEvent 语义变复杂。
- **在 `CheckBulletCollision(...)` 内临时发起任意时点查询**：能工作，但不如在推进路径处做 sweep 更物理正确。

### 6. IPC 技术优先使用命名管道，回退可接受本地 loopback TCP

**Decision**
- Windows 开发和部署环境下，首选 Named Pipe 作为本机 IPC。
- 若实现或调试成本过高，可接受本地 loopback TCP 作为首版替代。

**Rationale**
- 命名管道在同机通信场景下延迟低、部署简单、语义明确。
- 本地 TCP 更通用，适合作为保底方案，但不作为首选。

**Alternatives considered**
- **MemoryMappedFile + 事件同步**：复杂度过高，不适合第一阶段。

## Risks / Trade-offs

- **[查询世界不同步]** → 必须固定帧内顺序为“主服推进 → 同步代理体 → 发起查询”，并且查询服不自行推进玩家状态。
- **[IPC 查询量爆炸]** → 第一阶段只让关键投射物和技能走 Unity 查询，普通轻量路径可暂时保留旧判定；同时协议拆层避免重复同步。
- **[查询服异常导致战斗不可用]** → 主服务器保留旧距离判定作为降级路径，并在房间/进程级别记录健康状态与告警。
- **[玩家代理碰撞体与客户端视觉不一致]** → 在服务端统一维护玩家代理体尺寸配置，避免命中盒在主服、查询服和客户端之间漂移。
- **[动态障碍状态遗漏]** → 第一阶段限制支持范围，先覆盖静态地图与已知必要的动态阻挡体；未纳入同步的动态对象不得参与权威遮挡判定。
- **[未来要做历史物理回放时设计受限]** → 当前协议与职责划分保留 frameId 和 battleId 语义，为后续引入历史场景快照预留接口，但不在首版实现。

## Migration Plan

1. 新增 Unity headless 查询工程或服务器专用启动配置，确保可在无图形模式下加载服务器场景与静态 collider。
2. 在查询服内建立玩家代理体与查询管理器，实现 `SyncPlayers`、`ProjectileCast`、`LineOfSight`、`OverlapSphere`。
3. 在主服务器中增加查询服务客户端、连接管理与健康检查，但初期不切流量。
4. 将当前帧中的玩家位置同步到查询服，并在日志模式下对比查询结果与旧判定，不直接影响扣血。
5. 先把 `TickServerBullets(...)` 的关键投射物切到查询结果驱动，再观察一致性与耗时。
6. 验证稳定后扩展到遮挡与范围技能查询；查询服异常时允许按房间或全局回退到旧路径。

- 动态障碍物在你们当前玩法中的数量和变化频率如何，是否会影响第一阶段只做静态地图的前提？
- 是否需要在主服务器中对查询耗时做硬预算和熔断，例如超过某阈值就自动降级？
- 第一阶段哪些投射物必须接入 `ProjectileCast`，哪些可以继续使用旧距离判定以控制查询量？
