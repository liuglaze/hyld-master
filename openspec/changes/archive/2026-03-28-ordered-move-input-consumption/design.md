## Context

当前服务端移动输入链路分为两段：接收侧在 `Server/Server/BattleController.Network.cs` 中把 `OperationID` 作为 `SyncFrameId` 写入 `dic_movementInputBuffer`，并按帧号排序；消费侧在 `Server/Server/Battle.cs` 的 `CollectAndBroadcastCurrentFrame()` 中从后往前扫描缓冲，选择“`SyncFrameId <= frameid + InputFutureLeadTolerance` 的最新输入”执行。这个实现已经具备“不等缺帧”的特性，但它的消费语义是“最新合法输入优先”，而不是“按序前进”。

在乱序网络下，这会出现两个与目标语义不一致的问题：
1. 较早但仍未消费的输入，可能被较新的合法输入越过；
2. 一旦较新输入先被消费，系统虽然事实上已经前进，但当前实现没有显式的“最后已消费移动帧”概念，迟到旧帧只是间接被移除，不利于稳定定义和验证。

本次设计只覆盖服务端移动输入消费语义，保持现有攻击输入链路不变。攻击仍使用 `AttackOperation.ClientFrameId`、超时拒绝、去重、历史位置回溯与追帧补偿；客户端的 `targetFrame`、`predicted_frameID`、`sync_frameID` 节奏控制语义也不在本次范围内。

## Goals / Non-Goals

**Goals:**
- 将服务端移动输入消费规则改为“按 `SyncFrameId` 升序消费”。
- 在缺失目标帧时不阻塞当前服务端帧推进，而是允许跳到更大的可消费帧继续前进。
- 一旦服务端已消费到某个更大移动帧，后续迟到的更旧输入必须被明确拒绝，避免乱序回灌。
- 明确缓冲区塞满时的淘汰策略，保证新输入可进入，同时不让缓冲长期堆积过多旧样本。
- 保留现有基于 `ClientAckedFrame` 的缓冲清理、未来帧容忍窗口、以及 `lastValidMove` 惯性兜底机制。
- 补充日志与验证点，使“按序消费 / 跳缺帧 / 满缓冲淘汰 / 拒绝迟到旧帧”可被直接观测。

**Non-Goals:**
- 不修改攻击输入的接收、去重、超时、回溯和追帧逻辑。
- 不改变客户端上传字段含义，不要求客户端发送 `targetFrame`。
- 不把移动输入改造成严格逐帧 lockstep；缺帧仍然允许跳过，不等待补齐。
- 不重构 `BattleLoop`、权威帧下发、位置推进公式或 UDP 传输层。

## Decisions

### 1) 为每个玩家引入显式的“最后已消费移动帧”进度
- **决策**：新增按 `battlePlayerId` 维护的 `lastConsumedMoveFrame`（命名可在实现阶段确定），初始化为 0。移动消费与迟到判断都以此为基准。
- **原因**：当前系统只有缓冲内容，没有显式消费边界，导致“已经前进到哪里”只能通过删除行为间接表达。引入显式进度后，才能稳定定义“31 已经无效，因为 32 已经消费过”。
- **备选**：
  - 继续依赖 `pendingInputs.RemoveRange(...)` 的副作用表达进度：语义隐含，难以验证（不选）。
  - 直接用 `ClientAckedFrame` 代替消费进度：`ClientAckedFrame` 是客户端确认的权威帧，不等于服务端已消费的移动输入进度（不选）。

### 2) 消费策略改为“优先 nextExpected，否则跳到最小可用合法帧”
- **决策**：对某玩家的缓冲区始终按 `SyncFrameId` 升序看待；每次消费时先寻找 `lastConsumedMoveFrame + 1`，若该帧存在且不超出 `frameid + InputFutureLeadTolerance`，则消费它；若不存在，则从小到大寻找第一个满足 `SyncFrameId > lastConsumedMoveFrame` 且 `SyncFrameId <= frameid + InputFutureLeadTolerance` 的输入并消费，表示“跳过缺失帧继续前进”。
- **原因**：这同时满足两个目标：
  1. 正常情况下严格按序前进；
  2. 缺帧时不等待，仍可持续推进。
- **备选**：
  - 继续“最新合法输入优先”：会跳过更多中间输入，不符合目标（不选）。
  - 必须等到 `nextExpected` 到达才消费：会重新引入卡顿与等待（不选）。

### 3) 迟到旧帧在接收阶段就显式拒绝
- **决策**：`UpdatePlayerOperation()` 在写入 `dic_movementInputBuffer` 前，先检查 `syncFrameId <= lastConsumedMoveFrame`，若成立则直接拒绝该移动输入，并记录拒绝日志；不再让这类输入进入缓冲区。
- **原因**：既然系统允许“跳过缺失帧继续前进”，那任何落后于已消费进度的包都必须被视为过时数据。接收阶段拒绝比等到消费阶段再忽略更清晰，也能减少缓冲污染。
- **备选**：
  - 允许入缓冲，等消费阶段忽略：会增加排序/清理复杂度，并弱化观测性（不选）。
  - 只依赖 `ClientAckedFrame` 清理：无法覆盖“服务器已消费但客户端尚未 ack”的迟到旧帧（不选）。

### 4) 满缓冲时淘汰最旧输入，保留更新的连续窗口
- **决策**：当 `dic_movementInputBuffer` 达到容量上限后，如果又来了一个新的、未被判定为迟到旧帧的输入，则淘汰当前缓冲里 `SyncFrameId` 最小的那个，保留较新的连续窗口。即用户举的例子：`31,32,33` 来了 `34` 后，结果应为 `32,33,34`。
- **原因**：本次整体方向就是“消费按序，但缓冲保留更新的未来样本”。如果满缓冲时反而踢掉最新输入，会让客户端刚产生的新控制意图进不来；而淘汰最旧值，既符合滑动窗口语义，也更符合不等待缺失帧的目标。
- **备选**：
  - 满缓冲时踢掉最新输入：会让新控制意图被拒之门外，不符合实时控制目标（不选）。
  - 满缓冲时不接收任何新输入：会让缓冲在乱序时僵住（不选）。

### 5) 保留未来帧容忍窗口，但它只决定“可消费上界”，不改变按序规则
- **决策**：继续使用 `InputFutureLeadTolerance` 作为服务端当前帧可接受的未来上界；任何候选输入只有在 `SyncFrameId <= frameid + InputFutureLeadTolerance` 时才可消费。但在这个合法集合里，仍按“nextExpected 优先，再取最小更大帧”的顺序选择。
- **原因**：窗口机制用于容忍客户端轻微超前，是现有动态追帧系统的一部分；本次要改的是消费顺序，而不是取消未来容忍。
- **备选**：
  - 彻底去掉未来帧容忍：会让轻微超前输入长期滞留，反而降低现网稳定性（不选）。
  - 在合法集合里仍取最大值：会退回当前实现（不选）。

### 6) 未找到可消费输入时继续沿用 `lastValidMove` 惯性兜底
- **决策**：若缓冲区中没有任何 `SyncFrameId > lastConsumedMoveFrame` 且不超过未来容忍上界的输入，则不前进消费进度，继续使用 `dic_lastValidMove` 作为短时惯性输入。
- **原因**：本次目标是改变消费语义，不是移除现有的抗抖动兜底。没有新输入时维持短时惯性，仍是当前系统避免停顿的必要行为。
- **备选**：
  - 没有可消费输入时直接归零移动：会放大网络波动带来的顿挫（不选）。
  - 没有可消费输入时强行推进消费进度：会伪造未收到的输入帧，破坏语义（不选）。

### 7) 增加四类关键日志，支持节点验证与联调
- **决策**：为移动输入链路补充至少四类日志：
  - `ACCEPT_IN_ORDER`：命中 `nextExpected` 正常按序消费；
  - `SKIP_GAP_ACCEPT`：缺失目标帧时跳到更大帧消费；
  - `REJECT_STALE`：收到 `syncFrameId <= lastConsumedMoveFrame` 的迟到旧帧并拒绝；
  - `EVICT_OLDEST_ON_FULL`：满缓冲时淘汰最旧输入并接纳新输入。
- **原因**：这四类日志正好对应本次改动最核心的四种行为，没有它们很难验证设计是否真正落地。
- **备选**：
  - 只保留现有通用日志：无法区分“按序前进”“满缓冲淘汰最旧”和“最新优先”是否被修正（不选）。

## Risks / Trade-offs

- **[Risk] 跳过缺失帧后，客户端极晚到达的旧输入被拒绝，可能让个别短时控制样本彻底失效** → 这是本次语义的有意取舍；通过保留 `lastValidMove` 惯性与未来容忍窗口降低体感突兀。
- **[Risk] 接收侧和消费侧同时引入消费进度，若状态更新顺序错误会导致误拒或重复消费** → 将进度更新集中在实际消费成功后的单一路径，并在接收侧只做只读判定。
- **[Risk] 现有 `InputBufferSize=8` 在高乱序场景下仍可能过小，导致尚未消费的较早合法帧被窗口裁掉** → 本次先保持现有窗口，依赖新增日志观察是否需要后续单独调参。
- **[Trade-off] 语义更强后，实现复杂度高于“最新合法输入优先”** → 复杂度增加是为了换取规则清晰、验证明确、行为更符合预期。

## Migration Plan

1. 在服务端战斗控制器中新增每玩家的移动消费进度字段，并在 `BeginBattle()` / `HandleBattleEnd()` 生命周期中初始化与清理。
2. 调整 `UpdatePlayerOperation()`：保留现有 ack 清理、排序与窗口限制，但在入缓冲前增加“迟到旧帧拒绝”判断与日志。
3. 调整 `CollectAndBroadcastCurrentFrame()`：将当前“从后往前取最新合法帧”改为“优先 nextExpected，否则取最小更大合法帧”；仅在成功消费时推进 `lastConsumedMoveFrame`。
4. 补充日志与最小验证脚本/手工复现步骤，覆盖按序、跳缺、拒绝迟到三种情况。
5. 若联调观察到窗口不足或跳缺策略过于激进，再作为后续独立调参项处理。

**回退策略**：若新语义引发明显倒退，可回退到当前“最新合法输入优先”实现；本次改动集中在 `Battle.cs` 与 `BattleController.Network.cs` 的移动输入路径，回退面可控。

## Open Questions

- `lastConsumedMoveFrame` 的初始值是否保持为 0 即可，还是需要与首帧/开战同步消息做更明确对齐？
- 发生 `SKIP_GAP_ACCEPT` 时，是否需要记录被跳过的目标帧号，便于后续日志统计缺帧率？
- 现有 `pendingInputs.RemoveAll(item => item.SyncFrameId <= clientAckedFrame - 2)` 是否仍需保持 `-2` 余量，还是应在本次语义稳定后再单独评估？
- 是否需要在客户端联调日志中补一条对应的观测点，以便把上传顺序与服务端消费顺序对照起来？
