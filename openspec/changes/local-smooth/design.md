## Context

当前联机战斗链路是“服务端权威 + 客户端等待指令后推进”，核心表现为：
- 战斗主循环固定帧（`frameTime = 0.066f`），客户端依赖服务端下发帧/操作推进。
- 现有实现是输入/操作同步与本地推进混合模型，但未系统化实现“预测 + 回滚和解”。
- 代码层主要落在：
  - `Assets/Scripts/Server/Manger/Battle`（`BattleManger`、`HYLDPlayerManger`、`HYLDBulletManger`）
  - `Assets/Scripts/Manger`（`CommandManger` 输入采集）
  - `Assets/Scripts/Server`（协议读取、UDP/TCP 管理）

约束：
- 服务端权威不可改变；客户端不能“确认最终结果”。
- 需兼容单机/联机并存，不能把单机路径一起重构。
- 当前并非全链路纯 lockstep（存在 float 与 Unity 行为），要按“有限确定性”设计。
- 历史目录（`OldServer`/`HYLD1.0`）默认不触碰，除非调用链明确依赖。

## Goals / Non-Goals

**Goals:**
- 在联机战斗中引入客户端预测，降低操作输入到画面反馈的等待感。
- 建立可控的回滚和解机制：收到权威帧后可检测偏差并重演后续输入。
- 明确客户端/服务端职责边界，保持服务端最终裁决。
- 在不大改协议与传输层的前提下完成能力补齐（优先复用帧号、`OperationID`）。
- 小步落地，避免一次性重构战斗、网络、UI 三层。

**Non-Goals:**
- 不将项目改造成全确定性纯 lockstep 架构。
- 不重写 TCP/UDP 栈，不替换现有协议体系。
- 不清理或迁移历史目录（`OldServer`、`HYLD1.0`）作为本次目标。
- 不在本阶段引入复杂反作弊方案（仅保持服务端权威约束）。

## Decisions

### 1) 保持“服务端权威 + 客户端预测”双层模型
- **决策**：客户端可先行模拟本地输入，服务端帧到达后做对账；最终状态以服务端为准。
- **原因**：既能改善手感，又不改变现有联机安全边界。
- **备选**：
  - 继续完全等待服务端：实现简单但手感差（不选）。
  - 客户端权威：冲突现有架构与安全模型（不选）。

### 2) 以“帧号 + OperationID”组织输入历史与状态快照
- **决策**：建立固定窗口缓存（环形缓冲），至少包含：`frameId`、`operationId`、本地输入、关键战斗状态快照。
- **原因**：回滚重演需要稳定锚点；仅存输入不足以恢复状态。
- **备选**：
  - 仅缓存输入：无法可靠回滚到一致起点（不选）。
  - 全量深拷贝所有对象：成本过高、风险大（不选）。

### 3) 回滚范围采用“最小必要集合”并分阶段扩展
- **决策**：首版优先覆盖本地玩家移动/朝向/基础攻击链路，逐步扩展到更复杂对象。
- **原因**：当前工程耦合高、确定性有限，先控制复杂度与回归面。
- **备选**：
  - 一次性覆盖全部战斗对象：改动面大、验证成本高（不选）。

### 4) 和解触发采用阈值策略（位置/朝向/关键状态）
- **决策**：当客户端预测与权威状态偏差超过阈值时触发回滚；未超阈值只做轻量校正。
- **原因**：避免每帧回滚带来的性能抖动。
- **备选**：
  - 每次权威帧都强制回滚：稳定但代价高（不选）。
  - 永不回滚仅插值：长期漂移不可控（不选）。

### 5) 和解后采用短窗平滑，避免视觉硬跳
- **决策**：逻辑层先完成权威一致，再对表现层做短窗平滑（位置/朝向）。
- **原因**：直接硬纠偏会出现明显“瞬移/抖动”。
- **备选**：
  - 只硬纠偏：体验差（不选）。
  - 长时间插值：会拉低实时性并掩盖真实状态（不选）。

### 6) 明确模块职责与状态流转
- **客户端职责**：输入采集、预测推进、缓存、权威对账、回滚重演、表现平滑。
- **服务端职责**：广播权威帧与操作结果，维持最终裁决。
- **关键状态流转**：
  1. `CommandManger` 采集输入并生成本地操作。
  2. 本地立即推进预测帧，并写入缓存（输入+快照）。
  3. 收到服务端权威帧后，按 `frameId/operationId` 对账。
  4. 若偏差超阈值：回滚到权威锚点，重放后续本地输入。
  5. 输出和解后的逻辑状态，表现层做短窗平滑。

### 7) 联机开关隔离：仅联机战斗启用
- **决策**：通过战斗模式标记/配置开关仅在联机路径启用预测回滚，单机路径保持原行为。
- **原因**：降低兼容性回归风险，便于问题回退。
- **备选**：
  - 单机联机共用同一预测回滚链路：收益小、风险大（不选）。

## Risks / Trade-offs

- [浮点与非确定性导致重演结果不稳定] → 先覆盖确定性较高链路；增加偏差监控与阈值保护。
- [缓存窗口过小导致无法回滚，过大导致内存升高] → 使用可配置窗口并做运行时统计调优。
- [高频回滚造成 CPU 峰值与卡顿] → 阈值触发 + 限频策略，优先减少不必要回滚。
- [旧代码耦合导致联机/单机互相污染] → 严格联机开关隔离，按调用链小步改动。
- [BattleID 映射/镜像逻辑错误引发和解错位] → 在回滚入口统一映射校验并补充日志。
- [协议字段不足以做精确对账] → 先复用现有帧号/OperationID；不足时在后续 specs 明确增量字段与兼容策略。

## Migration Plan

1. **阶段 0（可观测性）**：先接入偏差采样与日志，不改现网行为，确认漂移来源与频率。
2. **阶段 1（预测）**：仅开启本地预测与输入缓存，不启用自动回滚；验证手感收益与稳定性。
3. **阶段 2（回滚和解）**：对关键链路启用阈值回滚重演，加入短窗平滑。
4. **阶段 3（扩展与收敛）**：按对象类型逐步扩展覆盖范围，完善边界场景（丢包、乱序、高延迟）。

**回退策略**：保留配置开关，出现严重问题可一键关闭预测/回滚，恢复“等待服务端推进”的旧路径。
