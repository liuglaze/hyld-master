## Why

当前客户端战斗推进主要依赖服务端下发指令，网络抖动会直接放大为本地输入延迟与角色“卡顿/回拉”。在现有 UDP 帧同步链路下补齐客户端预测与回滚和解，可以在不改变服务端权威性的前提下显著改善操作手感，并降低高延迟场景下的可感知不同步。

## What Changes

- 新增客户端本地预测：玩家本地输入在发送到服务端后立即驱动本地模拟，不再完全等待服务端回包后再执行。
- 新增输入历史与状态快照缓存：按帧/操作序号记录本地输入与关键战斗状态，用于后续回滚重演。
- 新增服务端权威对账与和解流程：收到服务端权威帧后，检测本地预测偏差；超阈值时执行“回滚到权威帧 + 重放后续本地输入”。
- 新增平滑纠偏策略：对位置/朝向等可视状态采用短窗平滑，避免硬纠偏造成明显抖动。
- 明确单机与联机路径边界：预测/回滚仅在联机战斗路径启用，单机逻辑保持兼容。
- 非目标：本变更不引入新的网络传输协议栈，不替换现有 TCP/UDP 职责分层。

## Capabilities

### New Capabilities
- `client-prediction-reconciliation`: 客户端在联机战斗中支持本地预测、权威校正、回滚重放与视觉平滑和解。

### Modified Capabilities
- （空）当前 `openspec/specs/` 下无既有 capability，暂无需要做 delta 的既有规格。

## Impact

- 需求类型：联机链路能力补齐（功能新增，带兼容性约束），不是纯修 bug。
- 主要受影响目录：
  - `Assets/Scripts/Server/Manger/Battle`（战斗帧推进、玩家/子弹模拟、回滚重演入口）
  - `Assets/Scripts/Manger`（输入采集与命令缓存）
  - `Assets/Scripts/Server`（协议字段读取与帧对账接入点）
- 风险标注：
  - **触碰战斗同步**：是（高风险）
  - **触碰协议结构**：原则上否（优先复用现有 `OperationID`/帧号）；若现有字段不足，需在后续设计阶段显式评估新增字段兼容性
  - **触碰全局状态**：是（`BattleData`/全局管理器中的帧号与状态缓存）
  - **触碰历史兼容目录（OldServer/HYLD1.0）**：否（仅在确有调用链依赖时再最小化触达）
- 预期外部影响：
  - 无对外 API 破坏性变更（暂不标记 BREAKING）
  - 手工验证重点新增：高延迟/丢包下的移动、攻击、命中反馈与回滚后状态一致性。
